# なぜ revu か

## 解きたかった問題

PR レビューは脳のリソースを食う割に、定型的な観点（命名、null 安全、テスト網羅、依存方向、セキュリティ）で時間が削られがちです。LLM はその「定型的な観点」を機械的に流せるはずで、Claude Code のような skill 駆動の環境ではすでに「PR を読んで draft レビューを書く」までは現実的にこなせます。

しかし生成物をそのまま GitHub にぶつけると次の問題が起きます。

- **質が荒い**: 一般論で押す指摘、誤検出、的外れな suggestion が混じる
- **責任の所在があいまい**: AI が書いたコメントを誰が承認したのか不明確
- **ノイズが残る**: 重要度の混ざったコメントが PR スレッドを埋め、PR 作者が大事な指摘を見落とす
- **ローカルで触れない**: GitHub に直接書き込むワークフローでは、推敲・並び替え・撤回が面倒

revu はこの「LLM の生成物を GitHub に流す前に人間がキュレーションする」中間レイヤーが必要だ、という認識から作りました。

## revu のスタンス

revu は **AI が下書きしたレビューを、人間が取捨選択して GitHub に投稿する道具** です。

- レビューの **生成** は Claude Code の review-pr skill が担当（revu 自身は LLM を内蔵しない）
- 生成物は `~/.revu/{owner}/{repo}/pr-{N}/{sha[:7]}/` 配下に YAML + Markdown で書き出される
- TUI (`revu open`) で 1 件ずつ accept / reject / edit
- 残ったものを `revu submit` で GitHub に投稿。投稿は人間の明示確認を要求

「人間が最終判断する」を仕組みで担保するのが目的です。レビューを **自動化** することは目指していません。

## 設計の根っこ

### ファイルベース

レビューは review.yml と Markdown 群です。$EDITOR でも `cat` でも `git diff` でも見られます。

- 生成 → 編集 → 投稿のどの段階でもエディタ・スクリプト・git で介入できる
- 投稿前にローカルでバージョン管理したり、チームに共有したりできる
- ブラックボックスな DB を持たないので、壊れても直せる

### SHA で履歴を分ける

`pr-{N}/{sha[:7]}/` という階層で、同じ PR の force-push に追従して別ディレクトリに書き出します。

- 古い SHA のレビューを残すか消すかは利用者の判断
- `revu prune` でクローズ／マージ済 PR のレビューだけを掃除できる

### レイヤード設定

設定（`config.toml`）・テンプレート（`templates/`）・コーディング規約（`guidelines`）はすべて 3 層に重なります。

1. `~/.config/revu/` — グローバル
2. `<repo>/.revu/` — プロジェクト共有（コミット推奨）
3. `<repo>/.revu-local/` — 個人ローカル（`.gitignore` 推奨）

「グローバルで自分の好み、プロジェクトでチームの規約、ローカルで個人の上書き」を 1 つの仕組みで扱えます。

### 投稿フローの安全装置

`revu submit` は次のときに止まります。

- `gh auth status` 失敗（未認証）
- review.yml の `head_sha` と PR の現 head が不一致（PR が更新されている）
- 既に `submitted_at` が記録されている（多重投稿防止）
- 確認プロンプトで `submit` 以外を入力した

「うっかり投稿」を仕組みで防ぐためのチェックです。

## 解決しないこと

- **レビューの自動化**: 人間が確認・編集・投稿します
- **LLM の内蔵**: Claude Code の skill が生成を担当します
- **GitHub 以外**: GitLab／Bitbucket／Forgejo は対象外です
- **同期コラボ**: 1 人がローカルでキュレーションするツールであって、複数人が同時編集する想定はしていません

## 似ているもの／違い

| ツール | revu との違い |
|---|---|
| GitHub PR コメントを直接書く | revu は AI の下書きを通すワンステップを足す。一発投稿の手軽さは失う代わりに、LLM の量を活かしつつ質を担保できる |
| コードレビュー bot（DeepSource、CodeRabbit など） | bot は自動投稿前提。revu は投稿前に人間が必ず介在する。bot の指摘を採用するかは PR 作者ではなくレビュアーが決める |
| Claude Code skill 単体 | skill 単体だと生成 → 投稿の中間レイヤーがなく、TUI での取捨選択や投稿前の確認がない |

## どう使うのが効くか

- **他人の PR を 1 日 5〜10 件レビューするチームメンバー**: 1 件あたりの初動を skill に任せ、人間は精度の高い指摘だけ拾う運用
- **OSS メンテナ**: コントリビュータの PR に最低限の観点（テスト・スタイル・セキュリティ）でフィードバックを返したいが、毎回手で書きたくない場合
- **コーディング規約をプロジェクトで揃えたいチーム**: `[review] guidelines` でコーディング規約・セキュリティチェックリストの Markdown を渡し、skill に観点として読ませる

## 何が起きていないとき revu は不要か

- レビューに費やす時間がそもそも短い／そこまで負担に感じていない
- AI 生成レビューを GitHub に直投稿する運用に既に十分満足している
- すべての PR を 1 行ずつ自分の目で読み直したい
- レビュー対象が GitHub 以外のプラットフォームにある

このどれかに当てはまるなら revu の追加レイヤーはコストの方が大きいかもしれません。
